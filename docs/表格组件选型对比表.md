# ZPP011 表格组件选型对比表

**起草人**：达析  
**日期**：2026-05-21  
**评审对象**：ZPP011偏差分析器 审核表格组件  
**评审目的**：决策是否替换现有 Tkinter Treeview，以解决大数据量卡顿、列宽自适应差、虚拟滚动缺失等问题

---

## 一、候选方案

| # | 方案 | 技术栈 | 定位 |
|---|------|--------|------|
| A | 优化现有 Tkinter Treeview | Tkinter 内置 | 低成本改进 |
| B | pandastable | Tkinter + pandas | 专用数据表格 |
| C | PyQt5 QTableView + QAbstractTableModel | PyQt5 | 工业级 UI 框架 |

---

## 二、详细对比

### 2.1 性能

| 维度 | A - 优化 Treeview | B - pandastable | C - QTableView |
|------|-----------------|-----------------|----------------|
| 虚拟滚动（懒加载） | ❌ 不支持，全量渲染 | ⚠️ 有限支持（分块加载） | ✅ 原生虚拟渲染 |
| 5000 行渲染速度 | 慢（3-8s） | 中等（1-3s） | 快（<0.5s） |
| 列排序性能 | ❌ 需手动重排全表 | ⚠️ 依赖 pandas 操作 | ✅ 模型层排序，不重绘 |
| 内存占用（5000行×20列） | 高（全量 Widget 节点） | 中 | 低（模型与视图分离） |

### 2.2 功能覆盖

| 功能 | A - 优化 Treeview | B - pandastable | C - QTableView |
|------|-----------------|-----------------|----------------|
| 行颜色标记 | ✅ 已实现（tag配色） | ✅ 内置 | ✅ delegate 自定义 |
| 单元格内编辑 | ❌ 需自制 Entry 覆盖层 | ✅ 双击编辑 | ✅ 原生 ItemDelegate |
| 右键菜单 | ✅ 已实现 | ✅ 内置有限右键 | ✅ 完全自定义 |
| 列宽拖拽 | ✅ 已有，但宽度不持久 | ✅ | ✅ |
| 列冻结（锁定首列） | ❌ | ❌ | ⚠️ 需自行实现（多 View） |
| 多列排序 | ❌ 需手动实现 | ⚠️ 单列 | ✅ QSortFilterProxyModel |
| 筛选联动（现有逻辑） | ✅ 已深度绑定 | ⚠️ 需重接口 | ✅ ProxyModel 天然支持 |
| 单元格 Tooltip | ❌ 需额外实现 | ❌ | ✅ 原生支持 |

### 2.3 开发成本

| 维度 | A - 优化 Treeview | B - pandastable | C - QTableView |
|------|-----------------|-----------------|----------------|
| 现有代码改动量 | 小（局部优化） | 中（替换组件，保留逻辑） | 大（整体 UI 框架迁移） |
| 学习成本 | 无 | 低（API 简单） | 中（Qt 信号槽机制） |
| 预估工时（仅表格部分） | 0.5天 | 1-2天 | 3-5天 |
| 与现有 Treeview 代码兼容 | ✅ 直接兼容 | ⚠️ 需适配列名、颜色接口 | ❌ 需重写 View 层 |
| 打包复杂度 | 低（tkinter 内置） | 中（增加 pandastable 依赖） | 高（PyQt5 + Qt 库 ~50MB） |

### 2.4 可维护性

| 维度 | A - 优化 Treeview | B - pandastable | C - QTableView |
|------|-----------------|-----------------|----------------|
| 社区活跃度 | 稳定（tkinter 官方） | ⚠️ 低维护（最近2年更新少） | ✅ 活跃（Qt 官方维护） |
| 文档完整性 | ✅ | ⚠️ 文档较少，靠源码摸索 | ✅ 完整官方文档 |
| Bug 风险 | 低（已知问题已处理） | 中（第三方库兼容问题） | 低（Qt 生态成熟） |
| 未来扩展（图表/复杂布局）| 差 | 差 | 好 |

### 2.5 用户体验

| 维度 | A - 优化 Treeview | B - pandastable | C - QTableView |
|------|-----------------|-----------------|----------------|
| 渲染清晰度 | 一般（Tkinter 字体渲染略粗糙） | 一般 | ✅ 高（HiDPI 支持好） |
| 交互流畅度（大数据量） | 差 | 中 | 好 |
| 与现有 UI 风格一致性 | ✅ 完全一致 | ⚠️ 风格有差异，需定制 | ❌ Qt 风格与 Tkinter 差异明显 |
| 移植到 Web/其他端 | ❌ | ❌ | ⚠️ Qt for WebAssembly（复杂） |

## 四、推荐方案与理由

### 近期（v38 范围内）：**推荐 A — 优化现有 Treeview**

**理由：**
1. ZPP011 当前日常数据量约 500-2000 行，优化 Treeview 后性能已可接受
2. 现有颜色标记、右键菜单、筛选联动等逻辑与 Treeview 深度耦合，切换成本高
3. 近期重心在审核记录存储机制重构（v37.44）和 MVP 迁移，不宜同期大改 UI 组件

**优化重点（A 方案具体落地）：**
- 添加分页加载（每次渲染 500 行，配合滚动事件动态追加）
- 缓存已渲染行的 tag 状态，避免全量刷新时重复着色
- 列宽配置持久化到 `%APPDATA%\ZPP011\config\column_width.json`

### 中期（v40+ 或数据量稳定超过3000行时）：**考虑 C — QTableView**

**理由：**
1. 若 MVP 重构完成、View 层彻底与逻辑解耦，迁移 Qt View 只需替换 View 层
2. QTableView + QSortFilterProxyModel 可原生支持多列筛选，替代现有 `filter_audit_data` 的 Python 逻辑
3. 未来若需要导出 PDF 报告、嵌入图表，Qt 生态有更好的支持

### 不推荐 B — pandastable

**理由：**
- 维护活跃度低（GitHub 最近2年仅有少量修补提交）
- 与现有颜色/筛选逻辑适配成本和 Qt 方案差不多，但长期收益远不如 Qt
- 打包后体积增加，没有显著的开发效率优势

## 五、决策门控条件

| 触发条件 | 动作 |
|---------|------|
| 单次加载数据 > 3000 行且卡顿 > 2秒 | 启动 QTableView 迁移专项 |
| MVP 迁移完成，View 层完全解耦 | 评估 QTableView 接入可行性 |
| pandastable 出现严重兼容 Bug | 直接跳过 B，视情况选 A 或 C |
